# Eval skeleton template — structural guidance

The eval skeleton is one file: `process-designer-output/evals/eval-set.md`. It contains twenty representative queries (sized to Iteration cost), a four-anchor scoring rubric per criterion, a results template the reader copies per run.

Evals are how the reader makes prompt and configuration changes empirical rather than vibes-based.

---

## Query mix surfaces what changes

Pull queries from the reader's use-case paragraph. **Recognisably from the reader's domain.** *"Summarise this document"* is decoration; *"summarise this incident postmortem into a one-paragraph executive briefing preserving named contributing factors"* is evaluable.

- ~10 **common tasks** — pass at high rates today; the eval catches regressions.
- ~6 **edge cases** — known-difficult conditions; fail today; improve over time.
- ~4 **aspirational tasks** — things the reader wants but currently doesn't get; failure targets.

Aim for ~12 of 20 passing at baseline — neither too easy nor too hard.

---

## Eval-set size scales with Iteration cost

| Iteration cost | Size |
| --- | --- |
| 1–2 (cheap) | Full twenty. |
| 3 (day-constraining) | Ten to fifteen. Drop the most expensive aspirational queries first. |
| 4 (real-world action) | Five to ten, carefully chosen. Eval cheap enough to run per change. |

For Iteration cost 4, suggest running a sample (3–5 queries) per change; the full set is reserved for major releases.

---

## Rubric criteria are orthogonal

Three default criteria, each on the four-anchor scale used throughout the skill:

- **Correctness.** Did the output do what was asked?
- **Calibration.** Was confidence proportional to the actual epistemic state? Penalise confident-wrong.
- **Domain fit.** Did the output read as native to the reader's domain, or generic?

Use-case quality dimensions from Stage 1 add criteria up to a maximum of five total. If two criteria always score the same, collapse them — orthogonality matters.

Anchor wording per criterion mirrors the reader's free-text register. *Plain*-register: *"AI got it mostly right, one small thing off."* Technical: *"Output correct with one minor deviation from spec."*

## Output file shape

```markdown
# Eval set for <use case name>

> Generated by `/process-designer` on YYYY-MM-DD. Dimension targets at generation: [list].
> Run after any prompt, skill, or configuration change. Track rates over time in `results/`.

## How to use

For each query: run the agent; capture the output; score with the rubric; record the score. Aggregate per criterion across the queries. Compare to baseline.
```

---

## Run discipline named in the eval-set preamble

- Run baseline at install time. Record.
- Make one change at a time. Re-run. Record deltas.
- Re-baseline when the configuration changes substantially.

Without the discipline, the skeleton is a list of queries with no measurement loop.
